# project-management/src/10-QA/PLANNING

QA gap analysis reports generated by scanning stories, sprints, and user flow documents for
testability, edge cases, and coverage gaps against `project-management/docs/QA-GUIDE.md`.

## Directory Tree

```text
project-management/src/10-QA/PLANNING/
├── CONTEXT.md                    ← this file
├── QA-REPORT-STORIES.md          ← per-story testability and edge case analysis  [created by planning workflow]
├── QA-REPORT-SPRINTS.md          ← per-sprint QA scope and coverage gaps         [created by planning workflow]
├── QA-REPORT-DATABASE.md         ← schema constraints, field validation, integrity rules [created by planning workflow]
└── QA-REPORT-USER-FLOWS.md       ← per-flow error states and boundary conditions [created by planning workflow]
```

## When to read these

- Before running `workflows/10-qa-checks/` on a new sprint
- When assessing the test scope of a story or sprint
- When preparing `QA-US###-<DESCRIPTION>.md` plan documents in `src/10-QA/`

## Reports

| File                      | Source scanned      | Coverage                                                                                                   |
| ------------------------- | ------------------- | ---------------------------------------------------------------------------------------------------------- |
| `QA-REPORT-STORIES.md`    | `src/01-STORIES/`   | Per-story: testability, acceptance criteria completeness, identified edge cases                            |
| `QA-REPORT-SPRINTS.md`    | `src/02-SPRINTS/`   | Per-sprint: QA tasks present, testing gaps, accessibility and GDPR test requirements                       |
| `QA-REPORT-DATABASE.md`   | `src/03-DATABASE/`  | Field constraints, null/unique rules, FK integrity, boundary values, and migration validation requirements |
| `QA-REPORT-USER-FLOWS.md` | `src/04-USER-FLOW/` | Per-flow: error states, boundary conditions, happy path vs failure path coverage                           |

These reports are generated by running the QA planning workflow against the current state of each
source folder. They have not yet been created — run `workflows/10-qa-checks/` to produce them.
