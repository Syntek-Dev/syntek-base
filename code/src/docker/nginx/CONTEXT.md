# code/src/docker/nginx — Nginx Configurations

Three Nginx config files covering dev, test, and the frontend static server used inside
staging and prod frontend containers.

> Staging and production do **not** have a dedicated Nginx Docker container — the server's
> own Nginx handles the reverse proxy. Only the static file server inside the frontend
> image uses these configs.

## Files

| File                   | Used by                           | Purpose                                                   |
| ---------------------- | --------------------------------- | --------------------------------------------------------- |
| `dev.conf`             | `nginx` service in dev stack      | Reverse proxy → backend + Next.js dev server              |
| `test.conf`            | `nginx` service in test stack     | Reverse proxy → backend-test + frontend-test              |
| `frontend-static.conf` | Frontend container (staging/prod) | Serves Next.js static export from `/usr/share/nginx/html` |

## Path routing (dev & test)

| Path                  | Upstream        | Notes                         |
| --------------------- | --------------- | ----------------------------- |
| `/graphql/`           | `backend:8000`  | WS upgrade for subscriptions  |
| `/admin/`             | `backend:8000`  |                               |
| `/static/`, `/media/` | `backend:8000`  |                               |
| `/health/`            | `backend:8000`  |                               |
| `/_next/webpack-hmr`  | `frontend:3000` | WS upgrade for HMR (dev only) |
| `/`                   | `frontend:3000` | WS upgrade for all other WS   |

`test.conf` uses `backend-test` and `frontend-test` as upstream names (matching
the service names in `docker-compose.test.yml`).

## X-Forwarded-Proto and dev/staging/prod parity

Both `dev.conf` and `test.conf` set `X-Forwarded-Proto: https` on **every** proxy request,
even though the Nginx container itself only listens on port 80. This is deliberate.

In staging and production, CF Tunnel terminates TLS at Cloudflare's edge and forwards requests
to the server over HTTP — but always with `X-Forwarded-Proto: https` in the header. Django reads
this header (via `SECURE_PROXY_SSL_HEADER`) to determine whether a request is secure.

By hardcoding `https` in dev and test, Django's view of the request is identical across all
four environments, eliminating an entire class of "works locally, breaks in production" failures.

### Failures this prevents

| Failure                             | Cause                                                 |
| ----------------------------------- | ----------------------------------------------------- |
| CSRF rejected                       | `CSRF_COOKIE_SECURE=True` — cookie not sent over HTTP |
| Session lost                        | `SESSION_COOKIE_SECURE=True` — same                   |
| `request.is_secure()` returns False | No `X-Forwarded-Proto` header                         |
| Infinite redirect loop              | `SECURE_SSL_REDIRECT=True`                            |

### Required Django setting

Add this to `config/settings/base.py` (applies to all environments):

```python
SECURE_PROXY_SSL_HEADER = ("HTTP_X_FORWARDED_PROTO", "https")
```

This tells Django to trust the `X-Forwarded-Proto` header from the upstream proxy.
Without it, Django ignores the header and `request.is_secure()` always returns `False`.

> **Do not set** `SECURE_SSL_REDIRECT = True` in `settings/local.py` or `settings/test.py` —
> the Nginx container listens on HTTP only, so a redirect to HTTPS would loop indefinitely.
> That setting is safe in `settings/staging.py` and `settings/production.py` only.

## frontend-static.conf

Serves a Next.js `output: 'export'` build. Key behaviours:

- `try_files $uri $uri.html $uri/index.html =404` handles clean URL routing
- `/_next/static/` assets served with `immutable` cache header (content-hashed filenames)
- Images and fonts cached for 24 hours
- Custom `404.html` error page (generated by Next.js during export)

## Server Nginx (staging / prod — not in this folder)

The server-level Nginx config is provisioned separately on the deployment server.
It should proxy:

```nginx
location ~ ^/(graphql|admin|static|media|health)/ {
    proxy_pass http://127.0.0.1:8000;
}

location / {
    proxy_pass http://127.0.0.1:3001;
}
```

SSL termination and CF Tunnel integration are handled at the server level.
